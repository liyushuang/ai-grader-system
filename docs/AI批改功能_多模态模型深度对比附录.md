## 一、主流多模态大模型全景

### 1.1 四大核心候选模型

| 模型 | 厂商 | 定位 | 多模态能力 | 中文优势 |
|------|------|------|-----------|---------|
| **GPT-4o** | OpenAI | 通用多模态旗舰 | 视觉+语音+文本一体 | 强（但非最强） |
| **Claude 3.5 Sonnet** | Anthropic | 逻辑推理之王 | 文档分析/视觉理解 | 中等 |
| **Gemini 2.0 Flash** | Google | 速度/成本优化 | 原生多模态架构 | **OCR能力突出** |
| **Qwen-VL-Max** (通义千问视觉) | 阿里巴巴 | 中文视觉专家 | 中文场景深度优化 | **最强** |

### 1.2 各模型详细画像

---

#### 🟢 GPT-4o（OpenAI）

**核心优势：**
- 业界公认的综合能力最强的通用多模态模型
- 响应速度快（端到端延迟低）
- 生态成熟，集成简单

**关键参数：**
- 价格：输入 $2.5/1M tokens, 输出 $10/1M tokens
- 图片限制：单次最多 16 张图
- 上下文：128K tokens
- **Grounding 能力**：❌ **不支持输出坐标**

**中文 OCR 表现：**
- 手写体中文识别率：约 85-90%（社区反馈）
- 对模糊/潦草手写体表现一般
- 古代文言文繁体字识别有困难

**适合场景：** 快速原型验证、非坐标依赖的纯文本批改

**不适合本项目的致命伤：**
> ❌ **无法输出 bounding box 坐标** — 无法直接在原图上渲染标注位置

---

#### 🔵 Claude 3.5 Sonnet（Anthropic）

**核心优势：**
- **逻辑推理和长文本理解能力业界第一**
- 文档分析能力极强（财报、论文、法律文书级别）
- 细节捕捉能力强，能发现细微错误

**关键参数：**
- 价格：输入 $3/1M tokens, 输出 $15/1M tokens（**最贵**）
- 图片限制：支持多图
- 上下文：200K tokens
- **Grounding 能力**：❌ **不支持输出坐标**

**中文 OCR 表现：**
- 印刷体中文：优秀
- 手写体中文：中等偏上（不如 Gemini）
- 推理判断质量：**顶级**（如果 OCR 准确，判断最可靠）

**适合场景：** 需要深度逻辑分析的批改任务（如作文评分、翻译逻辑校验）

**局限：**
> ❌ **价格最高** — 大规模使用成本压力巨大  
> ❌ **无坐标输出** — 同样无法直接定位错误位置

---

#### 🟡 Gemini 2.0 Flash（Google）—— ⭐ OCR 黑马

**核心优势：**
- **社区公认的手写中文 OCR 能力最强的大模型**（甚至超过 GPT-4o 和 Claude）
- **价格极低**：Flash 版本几乎免费
- 原生多模态架构（非后拼接），视觉理解更自然
- 支持超长上下文（1M+ tokens）

**关键参数：**
- 价格：**$0.075/1M tokens input, $0.30/1M tokens output**（是 GPT-4o 的 **1/37**）
- 图片限制：支持大量图片（取决于上下文窗口）
- 上下文：**1M tokens**
- **Grounding 能力**：⚠️ 有限支持（可通过特殊 prompt 引导输出相对坐标，但非原生 Grounding）

**中文 OCR 表现（重点）：**
- 手写体中文识别率：**约 90-95%**（社区 benchmark 反馈最佳）
- 对潦草/模糊/光照不均的学生作业照片容忍度高
- 古文繁体/异体字识别能力较好
- 支持 Systran 翻译增强层（可选）

**为什么它是"黑马"？**
```
Gemini 2.0 Flash 的OCR能力来源：
1. Google 内部集成了大量 Tesseract + 自研 OCR 技术积累
2. 训练数据包含 Google Books 的海量扫描古籍
3. 多模态原生架构让"看图识字"比"先OCR再理解"更准确
4. Flash 版本针对速度优化，延迟可接受
```

**适合场景：** 
- ✅ **本项目首选候选** — 性价比最优的 OCR + 理解一体化方案
- 大批量学生作业图片处理
- 成本敏感的 PoC / MVP 阶段

**风险点：**
- ⚠️ Grounding 非原生支持，需额外工程方案实现坐标映射
- 国内访问需要网络环境配置

---

#### 🔴 Qwen-VL-Max（阿里通义千问视觉）—— ⭐ 中文场景之王

**核心优势：**
- **唯一支持中文 Grounding（区域定位）的多模态大模型**
- DocVQA（文档视觉问答）benchmark 得分 **92.5%**（GPT-4V 仅 88.4%）
- 中文场景深度优化，古文/文言文理解能力强
- 国内服务，无网络访问问题

**关键参数：**
- 价格：约 ¥0.8/千次调用（按量计费，具体以 DashScope 官方为准）
- 图片限制：支持多图
- **Grounding 能力**：✅ **原生支持 bounding box 输出**

**中文 OCR 表现：**
- 手写体中文识别率：约 88-92%
- 文言文/古文理解：**业界最佳**（训练数据包含大量中文古籍）
- 标点符号、竖排文本、繁简转换：优秀

**独特杀手锏：Grounding 能力演示**

Qwen-VL-Max 可以直接输出带坐标的结构化结果：
```json
{
  "output": {
    "text": "该处翻译错误：'伐竹取道'应译为'砍伐竹林开辟道路'",
    "bounding_box": [x1, y1, x2, y2],  // 错误位置的像素坐标
    "label": "translation_error"
  }
}
```
这意味着：
- ✅ **无需独立 OCR 步骤即可获取文字位置**
- ✅ **可直接用于原图标注渲染**
- ✅ **端到端流程大幅简化**

**适合场景：**
- ✅ **本项目强力候选（排名第2）** — 如果需要坐标输出，这是唯一选择
- 中文教育场景、古籍文档处理
- 需要"圈出错误位置"的批改需求

**局限：**
- 国际 benchmark 表现略逊于 GPT-4o/Claude（但在中文领域反超）
- 复杂逻辑推理能力不如 Claude
- 生态系统不如 OpenAI 成熟

## 二、百度智能作业批改 API 详解

### 2.1 产品概述

百度 AI 开放平台的 **"智能作业批改"** 接口是面向 K12 教育场景的预包装解决方案。

### 2.2 能力覆盖

| 维度 | 支持情况 |
|------|---------|
| **适用年级** | 小学、初中、高中 |
| **学科覆盖** | 语文（含文言文）、数学、英语、物理、化学、生物 |
| **语文子类型** | ✅ 文言文翻译、✅ 作文评分、✅ 默写填空、✅ 阅读理解 |
| **输入格式** | 学生作业照片（JPG/PNG） |
| **输出格式** | **直接返回批改后的图片**（带红笔标注）+ 结构化 JSON |
| **API 类型** | 异步接口（提交→轮询结果，耗时 5-10 秒） |

### 2.3 工作原理

```
用户提交作业图片 → 百度服务器处理：
    ├── ① 内置 OCR（百度自研，针对 K12 优化）
    ├── ② 内置答案匹配（标准答案库 or 用户上传参考答案）
    ├── ③ 内置批改引擎（规则 + 模型混合）
    ├── ④ 内置图像渲染（模拟红笔批注效果）
    └── 返回：批改完成图 + 批改详情 JSON
```

### 2.4 核心优势

| 优势 | 说明 |
|------|------|
| **开箱即用** | 无需自己搭建任何 pipeline，调 API 就完事 |
| **可视化输出** | 直接返回带标注的图片，满足"班主任看图"需求 |
| **K12 场景深耕** | 百度在教育行业多年积累，对错别字/漏字/多字等常见错误模式覆盖全 |
| **稳定性保障** | 企业级 SLA，适合生产环境 |
| **无需 ML 团队** | 不需要懂模型，产品经理也能对接 |

### 2.5 核心劣势

| 劣势 | 说明 |
|------|------|
| **黑盒系统** | 不知道内部用了什么模型/规则，无法调试优化 |
| **定制化困难** | 只能在百度允许的范围内调整参数（如严格度等级） |
| **批改标准不可控** | 无法注入自定义的教研组批改规则（如国庆老师的标准） |
| **文言文支持未知** | 官方文档未明确说明文言文翻译批改的具体效果 |
| **数据隐私** | 学生作业图片上传至百度服务器 |
| **成本不透明** | 按次数计费，大批量使用费用可能高于自建方案 |
| **依赖第三方** | 服务变更/下线/涨价都无法控制 |

### 3.2 关键差异总结

#### 差异1：封装层级不同

```
百度 API（高度封装）：
  用户 → [黑盒: OCR+批改+渲染] → 批改完成图
       ↑ 你只能调参数，不能改逻辑

多模态LLM（低封装，高自由）：
  用户 → [模型: 看图+理解+判断] → 文字评判结果 → [自建: 渲染] → 根据你的规则输出
       ↑ 每一步都可控制、可替换、可调试
```

#### 差异2：批改标准的可控性

| 方案 | 注入自定义批改规则的方式 |
|------|------------------------|
| 百度 API | ❌ 只能选择预设的严格度等级（宽松/标准/严格） |
| GPT-4o / Claude / Gemini | ✅ 通过 System Prompt 注入详细规则（如："'伐'字必须译为'砍伐'，译为'攻打'扣1分"） |
| Qwen-VL-Max | ✅ 同上 + 还能结合 few-shot 示例增强 |

**对本项目的影响：**
> 国庆老师的批改标准文档（~100页/篇）中的细节规则，只能在多模态 LLM 方案中完整落地。百度 API 无法做到"某字的某个义项必须对应某个译文"这种细粒度控制。

#### 差异3：输出格式差异

| 输出需求 | 百度 API | GPT-4o/Claude/Gemini | Qwen-VL-Max |
|----------|---------|---------------------|-------------|
| 纯文本批改意见 | ✅ JSON | ✅ 文字回复 | ✅ 文字回复 |
| 错误位置（文字描述） | ✅ JSON | ✅ 文字描述 | ✅ 文字描述 |
| 错误位置（像素坐标） | ✅ 内部使用 | ❌ | ✅ **bounding box** |
| 带标注的原图 | ✅ **直接返回** | ❌ 需自建渲染 | ⚠️ 需简单渲染 |

#### 差异4：成本结构差异

**场景假设**：日均批改 1000 份作业，每份 1 张图

| 方案 | 日均成本（估算） | 月成本 | 备注 |
|------|----------------|--------|------|
| 百度 API | ¥50-200 | ¥1500-6000 | 按调用次数，具体价格需咨询 |
| GPT-4o | ~$15-30 | ~$450-900 | 按 token 计费 |
| Claude 3.5S | ~$25-50 | ~$750-1500 | **最贵** |
| **Gemini 2.0F** | **~$0.3-0.8** | **~$9-24** | **极具竞争力** |
| Qwen-VL-Max | ~¥8-20 | ¥240-600 | 国内服务定价 |

## 四、场景化选型建议

### 4.1 针对「文言文翻译批改」场景的排序

根据本项目特点（文言文优先、需要坐标定位、PoC阶段、周三Demodeadline），综合推荐排序：

#### 🥇 第一选择：Gemini 2.0 Flash

```
推荐理由：
✅ OCR能力最强（手写中文识别率最高）
✅ 成本几乎可以忽略不计（PoC阶段几乎免费）
✅ 速度快（Flash版本优化了延迟）
✅ 可通过 Prompt 注入国庆老师的批改细则
⚠️ 唯一短板：需自建坐标映射方案（但PoC阶段可先用文字描述替代）

适用条件：能解决网络访问问题（或使用代理）
```

#### 🥈 第二选择：Qwen-VL-Max

```
推荐理由：
✅ 唯一原生支持 Grounding（直接输出坐标）— 架构上最匹配
✅ 中文/文言文理解能力最强
✅ 国内直接可用，无网络障碍
✅ DocVQA得分业界最高
⚠️ 成本比 Gemini 高（但仍远低于 GPT-4o/Claude）

适用条件：如果"坐标输出"是硬性需求或国内部署是约束
```

#### 🥉 第三选择（基线对照）：百度 API

```
推荐理由：
✅ 开箱即用，集成最快（可能半天内跑通）
✅ 直接输出批改完成图（满足 Demo 演示需求）
✅ 适合作为 PoC 的 baseline 对照组
❌ 定制化能力弱（无法注入精细批改规则）

适用条件：快速出 Demo 给领导看，或者作为性能对照基准
```

#### ❌ 不推荐：GPT-4o / Claude 3.5 Sonnet（当前阶段）

```
原因：
❌ GPT-4o：无坐标输出 + 中等OCR + 不便宜 — 没有明显优势
❌ Claude：最贵 + 无坐标 + 国内难访问 — 性价比最低

除非未来需要复杂逻辑推理（如作文深评），否则暂不考虑
```

### 4.2 混合方案可能性（进阶）

如果时间允许，可以考虑 **"OCR 分离 + 判断融合"** 的混合架构：

```
方案 D（混合架构）：
  图片 → Qwen-VL-Max（利用 Grounding 获取文字+坐标）
       ↓
  结构化中间产物（文字 + bbox + 置信度）
       ↓
  分支A：简单错误 → 规则引擎快速判定（错别字/漏字/多字）
  分支B：复杂语义 → Gemini 2.0 Flash 深度判断（翻译准确性/信达雅）
       ↓
  合并结果 → 渲染到原图 → 输出
```

**优点：** 结合两者所长（Qwen的坐标 + Gemini的性价比+判断力）
**缺点：** 复杂度增加，PoC阶段不建议采用

## 五、PoC验证建议（更新版）

基于以上对比，建议周三 Demo 采用 **三轨并行验证**：

### Track A：Gemini 2.0 Flash 端到端（主力验证）

```
验证目标：OCR准确性 + 批改判断质量 + 成本可行性
输入：10-20张《小石潭记》学生翻译作业照片
流程：图片 → Gemini → Prompt引导输出类JSON评判 → 人工审核准确率
度量：识别准确率 / 批改一致性 / 响应时间 / token消耗
预期：OCR准确率>90%，批改判断与人工一致率>80%
```

### Track B：Qwen-VL-Max Grounding 验证（坐标能力验证）

```
验证目标：bounding box 输出的准确性和可用性
输入：5-10张同上样本
流程：图片 → Qwen-VL-Max → 输出带坐标的评判 → 验证坐标是否精准框选错误
度量：坐标IoU>0.7的比例 / 坐标输出成功率
预期：70%以上的错误能被准确定位
```

### Track C：百度 API 基线对照（效率参照）

```
验证目标：开箱即用的效果基线和出图能力
输入：5-10张同上样本
流程：图片 → 百度API异步调用 → 获取批改完成图 + JSON → 人工评价
度量：端到端耗时 / 批改完成图美观度 / 批改准确率
预期：跑通流程，记录效果作为baseline
```
