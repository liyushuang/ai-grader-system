## 一、会议讨论方案回顾

### 原方案流程（插件 Demo 时期）

```
学生拍照上传图片
    ↓
① OCR 识别（开源手写体识别）→ 获取文字 + 坐标
    ↓
② 匹配课文知识库（原文/译文/重难点/易错字）
    ↓
③ AI 批改判断（GPT）→ 输出结构化 JSON（标注类型+内容+坐标）
    ↓
④ 标注渲染到原图（波浪线/横线/圈/点睛句标记）
    ↓
⑤ 输出批改完成图 → 班主任确认 → 推送家长
```

### 关键约束（会议明确）

- **范围**：先做 PoC 验证可行性，以《小石潭记》为验证数据
- **APP 改造**：本次不考虑（暂不改造上传端为文档扫描模式）
- **目标交付时间**：周三（7月9日）产出 Demo
- **批改标准**：有文档可供总结提炼规则
- **作业类型优先级**：文言文翻译 > 作文 > 笔记

## 二、原方案的优劣势分析

### ✅ 优势

| 维度 | 说明 |
|------|------|
| **思路清晰** | OCR → 知识库匹配 → AI 判断 → 渲染标注，流水线分工明确 |
| **有历史验证** | 插件 Demo 已跑通全流程，核心路径已被验证可行 |
| **输出直观** | 最终产出"批改完成的图"，与现有班主任工作流一致 |
| **可控性强** | 结构化 JSON 中间产物，每一步可调试、可替换 |
| **知识库驱动** | 逐篇课文精细规则，批改质量有保障 |

### ⚠️ 劣势与风险

| 风险点 | 严重程度 | 说明 |
|--------|----------|------|
| **OCR 准确率瓶颈** | 🔴 致命 | 学生拍照质量差（斜拍/模糊/光线不足），开源 OCR 识别率低；且 APP 不做文档扫描改造的情况下，这个问题无法从源头解决 |
| **坐标偏移累积误差** | 🔴 高 | OCR 坐标不准 → 标注位置偏移 → 批改图看起来不专业，用户体验差 |
| **链路长、排障难** | 🟡 中 | 5 个环节串联，任一环节出错都影响最终结果，PoC 阶段调试成本高 |
| **知识库建设成本高** | 🟡 中 | 每篇 ~100 页精细规则，需教研深度参与 |
| **依赖外部模型** | 🟡 中 | GPT 效果好但成本高、合规性待评估；千问效果差 |

## 三、替代方案对比评估

### 方案 A：原方案（OCR + 知识库 + GPT 分步流水线）

```
图片 → OCR(文字+坐标) → 知识库匹配 → GPT批改(JSON) → 渲染标注 → 批改图
```

| 维度 | 评分 |
|------|------|
| **实现复杂度** | ⭐⭐⭐⭐ 高（5个模块串接） |
| **PoC 开发周期** | 3-5 天 |
| **批改准确度** | ⭐⭐⭐ 受限于 OCR 质量 |
| **标注精确度** | ⭐⭐⭐ 取决于 OCR 坐标精度 |
| **扩展性** | ⭐⭐⭐⭐ 好（模块化可替换） |
| **成本** | 低（开源 OCR）~ 中（GPT API） |

**适用场景**：正式产品化阶段，APP 上传已优化、对标注精度要求高

---

### 方案 B：多模态端到端方案（GPT-4o 直接理解图片）

```
图片 + 课文知识库(Prompt注入) → GPT-4o 多模态直接批改 → 输出批改结果
```

**核心思路变化**：
- **跳过独立 OCR 步骤**：让 GPT-4o 直接"看图"，利用其内置的多模态视觉能力识别手写文字
- **跳过坐标渲染**：GPT-4o 直接输出带标注的图片或结构化批改意见
- **Prompt 注入知识库**：将《小石潭记》的原文、译文、易错字、点睛句等规则写入 System Prompt

| 维度 | 评分 |
|------|------|
| **实现复杂度** | ⭐⭐ 低（1-2 个 API 调用） |
| **PoC 开发周期** | **1-2 天** |
| **批改准确度** | ⭐⭐⭐⭐ GPT-4o 视觉能力较强 |
| **标注精确度** | ⭐⭐ 无法精确定位坐标（除非额外处理） |
| **扩展性** | ⭐⭐⭐ 中（依赖模型能力演进） |
| **成本** | 中-高（GPT-4o API 较贵） |

**关键优势**：
1. **开发速度极快** — PoC 可能 1-2 天就能出效果，完美匹配周三 Demo 的紧急时间线
2. **绕过 OCR 瓶颈** — GPT-4o 的多模态理解能力在手写中文上表现不错，且不需要单独的坐标系统
3. **端到端调试简单** — 只有一个调用环节，问题定位容易
4. **天然支持"脏数据"** — 对斜拍、模糊等图片的容忍度比传统 OCR 更高

**关键局限**：
1. **无法精确渲染标注到原图坐标** — 如果必须产出"带红圈红线的批改图"，仍需要额外处理
2. **成本较高** — GPT-4o 比 GPT-3.5/千问贵很多
3. **输出格式控制** — 需要精心设计 Prompt 来确保输出结构化的 JSON

**PoC 变通方案**：如果 GPT-4o 不能直接输出标注图，可以采用混合模式 — 用 GPT-4o 做**批改判断**（文字级），再用轻量级工具做**可视化呈现**。

### 方案 C：商用 OCR API + 大模型混合方案

```
图片 → 商用OCR(百度手写作文API) → 结构化文本+坐标 → GPT/国产大模型 批改 → 渲染标注
```

**核心思路**：用百度/讯飞的成熟教育 OCR 替代开源方案，解决手写体识别问题。

| 维度 | 评分 |
|------|------|
| **实现复杂度** | ⭐⭐⭐ 中（3 个模块） |
| **PoC 开发周期** | 2-3 天 |
| **批改准确度** | ⭐⭐⭐⭐ 百度教育 OCR 经过专门优化 |
| **标注精确度** | ⭐⭐⭐⭐ 商用 API 提供字/行/段落级坐标 |
| **扩展性** | ⭐⭐⭐⭐ 好 |
| **成本** | 低-中（百度 OCR 有免费额度） |

**参考产品**：
- **百度智能作业批改 API**：K12 全学科，基于多模态大模型，直接支持批改输出
- **百度手写作文识别（多模态）**：专门针对手写作文优化，过滤涂改痕迹，输出字/行/段落坐标
- **讯飞星火智能批改**：硬件+软件整体方案，2000+学校落地

**特别值得注意的是百度"智能作业批改"API**：
> 基于**多模态大模型**能力，支持拍照/图片上传，**精准输出每道题批改结果**，同步提供解析，**支持结果原图渲染**。

这意味着百度可能已经封装好了「OCR + 批改判断 + 原图渲染」的全链条！值得优先测试。

### 方案 D：通古大模型 + RAG 方案（文言文专用）

```
图片 → OCR → 通古大模型(7B, 文言文专用) + RAG检索课文知识库 → 批改输出
```

**核心思路**：使用华南理工开源的**通古大模型**（基于百川2-7B，24亿字古籍语料训练），专攻文言文翻译批改。

| 维度 | 评分 |
|------|------|
| **实现复杂度** | ⭐⭐⭐⭐ 高（需部署私有模型+RAG） |
| **PoC 开发周期** | 3-5 天 |
| **批改准确度** | ⭐⭐⭐⭐ 文言文领域专业度高 |
| **标注精确度** | ⭐⭐⭐ 需配合其他组件 |
| **扩展性** | ⭐⭐ 仅限文言文场景 |
| **成本** | 低（可本地部署，无 API 费用） |

**优势**：文言文领域最专业的开源模型，支持文白互译、断句、纠错
**局限**：仅适用于文言文；7B 模型部署需 GPU 资源；不支持多模态（不能直接看图）；社区活跃度未知

### 4.3 与百度智能作业批改 API 的关键差异

这是一个关键的决策点：**到底用「通用多模态 LLM」还是「百度专门做的教育批改 API」？**

#### 差异一：能力封装层级不同

```
┌─────────────────────────────────────────────────────┐
│              百度智能作业批改 API                      │
│  ┌─────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │
│  │ 图片预处理 │→│ OCR识别  │→│ 批改判断  │→│ 原图渲染 │ │
│  │(矫正/增强)│ │(教育优化) │ │(多模态LLM)│ │(带标注)  │ │
│  └─────────┘ └──────────┘ └──────────┘ └─────────┘ │
│           一站式 API 调用，开箱即用                    │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│              多模态 LLM 方案（以 GPT-4o 为例）         │
│  ┌──────────────────────────────────────────────┐   │
│  │        图片 + 知识库Prompt → GPT-4o → JSON   │   │
│  │         "看图并按规则批改这段文言文翻译"       │   │
│  └──────────────────────────────────────────────┘   │
│           只管批改判断，OCR/渲染需自建               │
└─────────────────────────────────────────────────────┘
```

**结论**：
- 百度 = **全包套餐**（连筷子都给你备好了）
- 多模态 LLM = **半成品食材**（菜很好但得自己炒）

#### 差异二：批改逻辑的可定制程度

| 维度 | 百度 API | 多模态 LLM |
|------|----------|-----------|
| **批改规则** | 固定（通用 K12 标准） | **完全自定义**（通过 Prompt 控制） |
| **文言文特殊需求** | ⚠️ 可能不支持 | ✅ 可以针对《小石潭记》写专属规则 |
| **批改风格调整** | ❌ 不可调 | ✅ 随时调整（严格/宽松/鼓励型等） |
| **输出格式** | 固定 JSON | 自由定义 JSON Schema |
| **新增题型支持** | 取决于百度更新 | ✅ 只要 Prompt 写得好就能支持 |

**关键洞察**：

> 你们的需求是 **「文言文翻译批改」** — 这是一个高度细分、非标准的场景。
>
> 百度的 API 主要覆盖的是 **选择题/填空题/计算题/英语/作文** 等标准 K12 题型。
> **「字字落实翻译的文言文批改」这种场景，百度大概率没有针对性优化。**
>
> 反而多模态 LLM + 自定义 Prompt，更能精准表达你们的批改标准。

#### 差异三：输出形态不同

| 维度 | 百度 API | 多模态 LLM |
|------|----------|-----------|
| **输出形式** | 带标注的图片 URL（cropUrl） | 结构化文本（JSON/Markdown） |
| **标注类型** | 对/错标识（0/1/2/3） | 自定义（波浪线/横线/圈/旁批等） |
| **视觉效果** | ✅ 即用（可直接展示给班主任） | ❌ 需自行开发渲染层 |
| **批改粒度** | 题目级 | ✅ 可精细到字词级 |
| **评语生成** | 仅错因描述（reason 字段） | ✅ 可生成完整评语/修改建议 |

**实际影响**：

> 如果 Demo 要求 **"拿出一张带红圈红线的批改图给领导看"** → 百度 API 更容易出效果
>
> 如果 Demo 目标是 **"证明 AI 能精准批改文言文翻译"** → 多模态 LLM 更能体现深度价值

#### 差异四：成本结构

| 场景 | 百度 API | GPT-4o | Gemini 2.0 Flash | Qwen-VL-Max |
|------|----------|--------|------------------|-------------|
| 单次批改成本 | ~¥0.03-0.10/次（估） | ~¥0.5-2.0/次 | ~¥0.001/次 | ~¥0.01-0.05/次 |
| 月1000次批改 | ¥30-100 | ¥500-2000 | ¥1 | ¥10-50 |
| PoC 验证阶段 | 几乎免费（赠送额度） | $5-20 | **免费** | **免费** |

1. **会议说"千问跟弱智一样"可能指的是旧版通义千问文本模型** — 但 **Qwen-VL-Max（视觉语言模型）是完全不同的产品**，它在文档理解和中文场景的基准测试中超越了 GPT-4V，值得重新评估
2. **最大的惊喜是 Gemini 2.0 Flash** — OCR 能力被社区评为"恐怖如斯"，价格几乎可以忽略不计，应该是本次 PoC 的 MVP 选择
3. **真正的分水岭在于"要不要坐标"**：
   - 不要坐标（PoC 阶段证明能力）→ **Gemini 2.0 Flash** 最优
   - 要坐标（产出标注图）→ **Qwen-VL-Max** 或 **百度 API** 更合适
4. **百度 API 的角色从"首选"调整为"基线对照"** — 它更擅长标准 K12 题型，文言文翻译批改这种细分场景可能需要自定义 LLM 方案
